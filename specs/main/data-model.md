# Data Model: Define Company Logistic Place Platform

**Version**: 1.0  
**Created**: 2026-04-10  
**Architecture**: Microservices with Database-per-Service Pattern

## Overview

The logistics platform follows a microservices architecture with strict database separation per service. Each service owns its data schema exclusively, and cross-service data access occurs only through REST APIs or domain events via Kafka.

## Service Schemas

### Company Service Schema: `company`

**Purpose**: Manages partner companies (shippers, carriers, or both)  
**Ownership**: Company Service exclusively

#### Companies Table: `company.companies`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | VARCHAR(36) | PRIMARY KEY, NOT NULL | UUID generated by application |
| `name` | VARCHAR(200) | NOT NULL, UNIQUE | Company display name |
| `type` | VARCHAR(10) | NOT NULL, CHECK | SHIPPER, CARRIER, or BOTH |
| `status` | VARCHAR(10) | NOT NULL, CHECK | ACTIVE, INACTIVE, PENDING |
| `tax_id` | VARCHAR(50) | NULL | Government tax identifier |
| `contact_name` | VARCHAR(200) | NULL | Primary contact person |
| `contact_email` | VARCHAR(200) | NULL | Contact email address |
| `contact_phone` | VARCHAR(50) | NULL | Contact phone number |
| `notes` | TEXT | NULL | Additional company notes |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Creation timestamp |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Last modification timestamp |

**Indexes**:
- `idx_companies_status` on `status` (filtering by operational status)
- `idx_companies_name_lower` on `LOWER(name)` (name search)

**Business Rules**:
- Company names must be unique across the platform
- Status lifecycle: PENDING → ACTIVE → INACTIVE (terminal states)
- Type defines logistics capabilities and determines calculation participation

---

### Site Service Schema: `site`

**Purpose**: Manages consumption sites (geographic inputs for barycenter calculations)  
**Ownership**: Site Service exclusively

#### Consumption Sites Table: `site.consumption_sites`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | VARCHAR(36) | PRIMARY KEY, NOT NULL | UUID generated by application |
| `company_id` | VARCHAR(36) | NOT NULL | Reference to company (logical FK) |
| `name` | VARCHAR(200) | NOT NULL | Site display name |
| `description` | TEXT | NULL | Site description and purpose |
| `latitude` | DOUBLE PRECISION | NOT NULL, CHECK (-90 to 90) | WGS-84 latitude coordinate |
| `longitude` | DOUBLE PRECISION | NOT NULL, CHECK (-180 to 180) | WGS-84 longitude coordinate |
| `weight_tons` | DOUBLE PRECISION | NOT NULL, CHECK (>= 0), DEFAULT 0.0 | Annual consumption volume |
| `address` | VARCHAR(500) | NULL | Physical street address |
| `city` | VARCHAR(200) | NULL | City name |
| `country` | VARCHAR(100) | NULL | Country name |
| `status` | VARCHAR(10) | NOT NULL, CHECK, DEFAULT 'ACTIVE' | ACTIVE or INACTIVE |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Creation timestamp |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Last modification timestamp |

**Indexes**:
- `idx_sites_company_id` on `company_id` (company-specific site listing)
- `idx_sites_company_status` on `(company_id, status)` (active sites for calculations)

**Business Rules**:
- Sites belong to exactly one company (no sharing)
- Only ACTIVE sites participate in barycenter calculations
- Geographic coordinates must be valid WGS-84 coordinates
- Weight represents annual consumption volume for calculation weighting

---

### Calculation Service Schema: `calculation`

**Purpose**: Stores barycenter calculation results and audit trail  
**Ownership**: Calculation Service exclusively

#### Logistics Centers Table: `calculation.logistics_centers`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | VARCHAR(36) | PRIMARY KEY, NOT NULL | UUID generated by application |
| `company_id` | VARCHAR(36) | NOT NULL | Reference to company (logical FK) |
| `optimal_latitude` | DOUBLE PRECISION | NOT NULL, CHECK (-90 to 90) | Calculated optimal latitude |
| `optimal_longitude` | DOUBLE PRECISION | NOT NULL, CHECK (-180 to 180) | Calculated optimal longitude |
| `total_weighted_tons` | DOUBLE PRECISION | NOT NULL, CHECK (>= 0), DEFAULT 0.0 | Sum of all input site weights |
| `algorithm_description` | VARCHAR(100) | NOT NULL | Algorithm used (Weighted, Weiszfeld, etc.) |
| `iteration_count` | INTEGER | NOT NULL, DEFAULT 1 | Iterations for convergence |
| `convergence_error_km` | DOUBLE PRECISION | NOT NULL, DEFAULT 0.0 | Final calculation accuracy |
| `status` | VARCHAR(15) | NOT NULL, CHECK, DEFAULT 'CANDIDATE' | Workflow status |
| `reviewer_notes` | TEXT | NULL | Manual review comments |
| `calculated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Calculation timestamp |
| `reviewed_at` | TIMESTAMPTZ | NULL | Review/approval timestamp |

**Indexes**:
- `idx_lc_company_id` on `company_id` (company-specific results)
- `idx_lc_status` on `status` (workflow filtering)

#### Logistics Center Sites Table: `calculation.logistics_center_sites`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `logistics_center_id` | VARCHAR(36) | PRIMARY KEY (composite), FK | Reference to calculation result |
| `site_id` | VARCHAR(36) | PRIMARY KEY (composite) | Reference to input site |
| `sort_order` | INTEGER | NOT NULL, DEFAULT 0 | Input ordering |

**Indexes**:
- `idx_lc_sites_center` on `logistics_center_id` (calculation inputs lookup)

**Business Rules**:
- Status workflow: CANDIDATE → APPROVED → CONFIRMED OR CANDIDATE → REJECTED
- Only APPROVED calculations can be CONFIRMED (implemented in business logic)
- Calculation inputs are immutable once created (audit trail preservation)
- Convergence error measures algorithm accuracy in kilometers

## Domain Events (Kafka Topics)

Cross-service communication uses domain events published to Kafka topics for eventual consistency.

### Company Events Topic: `logistics.company-events`

- **CompanyCreatedEvent**: New company registration
- **CompanyStatusChangedEvent**: Status lifecycle changes

### Site Events Topic: `logistics.site-events` 

- **SiteAddedEvent**: New consumption site created
- **SiteUpdatedEvent**: Site location or weight modified  
- **SiteRemovedEvent**: Site deactivated/deleted

### Calculation Events Topic: `logistics.calculation-events`

- **BarycentreCalculatedEvent**: New calculation result available

**Event Structure**: All events implement `DomainEvent` interface with:
- `eventId` (UUID)
- `eventType` (polymorphic discriminator)
- `occurredAt` (timestamp)
- `aggregateId` (entity ID that triggered event)

## Data Relationships

### Logical Foreign Keys

The platform uses logical foreign keys (ID references) instead of database foreign keys to maintain service independence:

```
Company (1) ──────── (N) ConsumptionSite
Company (1) ──────── (N) LogisticsCenter  
LogisticsCenter (1) ─ (N) LogisticsCenterSites ── (1) Site (logical)
```

### Cross-Service Data Access

- **Company → Sites**: Site Service queries by `company_id`
- **Company → Calculations**: Calculation Service queries by `company_id`  
- **Calculations → Sites**: Calculation Service calls Site Service REST API
- **Dashboard Aggregation**: Dashboard Service aggregates via REST APIs

---

## Data Consistency Patterns

### Eventual Consistency

Changes propagate through domain events:
1. Company Service publishes `CompanyCreatedEvent`
2. Site Service and Calculation Service update local read models
3. Dashboard Service refreshes aggregated views

### Transactional Boundaries

Each service maintains strong consistency within its own database:
- Company Service: Company aggregate consistency
- Site Service: Site lifecycle and geographic validation
- Calculation Service: Calculation results and audit trail

### Compensation Patterns

Failed cross-service operations use compensation:
- If barycenter calculation fails, sites remain in previous state
- If company deactivation fails in downstream services, eventually consistent reconciliation occurs

## Performance Considerations

### Database Scaling

- **Read Replicas**: Each service can use read replicas for query scaling
- **Partitioning**: Large datasets can be partitioned by `company_id`
- **Caching**: Redis caching for frequently accessed calculation results

### Query Patterns

**Company Service**:
- List active companies: `WHERE status = 'ACTIVE'`
- Company name search: `WHERE LOWER(name) LIKE ?`

**Site Service**:
- Active sites for calculation: `WHERE company_id = ? AND status = 'ACTIVE'`
- Geographic proximity: PostGIS extensions for spatial queries (future)

**Calculation Service**:
- Latest results: `WHERE company_id = ? ORDER BY calculated_at DESC`
- Approved calculations: `WHERE status = 'APPROVED' OR status = 'CONFIRMED'`

## Security and Compliance

### Data Classification

- **Public**: Company names, general site locations
- **Internal**: Contact information, detailed addresses
- **Confidential**: Calculation algorithms, business volumes
- **Restricted**: Audit trails, compliance data

### Access Control

- Service-to-service: JWT authentication with service accounts
- Database access: Each service has dedicated database user with schema-only access
- API access: Role-based access control (RBAC) with company-scoped data isolation

### Audit Requirements

All tables include audit timestamps (`created_at`, `updated_at`)
- Calculation Service maintains full audit trail of results
- Domain events provide cross-service audit capability
- Database-level triggers can log sensitive data changes

## Migration and Evolution
